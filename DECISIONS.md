# Decisions Register (append-only)

## [2026-07-06] Live-agent initiative — three-phase direction (LIVE.1 = Phase A)
**Context:** User wants LifeDash to shift from post-hoc pipeline (record → review later) to a live experience: an AI partner during recording, evolving toward Graphify-style structured knowledge (entities + typed edges + confidence tags, queryable by the agent).
**Decision:** Three sequential phases: A) Live Assistant (drawer with live transcript + on-demand agent chat with tools), B) ambient auto-triage feed (proactive suggestions, one-tap accept), C) knowledge-graph substrate (entity/edge tables in PGlite, extraction post-meeting first). LIVE.1 covers A only.
**Alternatives:** Building the knowledge graph first (rejected — most experimental, needs the live surface to be useful); one mega-plan for all three (rejected — exceeds 6-task limit, bundles unrelated smoke tests).
**Rationale:** Each phase validates the next; A reuses proven infra (streaming transcription, card-agent tool loop) so it's the cheapest slice that changes the product's feel.

## [2026-07-06] "Live Assistant" naming (not "Copilot")
**Context:** User asked whether "copilot" meant Microsoft Copilot — the term is now trademark-loaded and could read as a Microsoft integration.
**Decision:** User-facing name "Live Assistant"; internal naming `meeting_agent` (schema, service, IPC), task type `live_assistant`.
**Alternatives:** "Meeting Partner", "Live Intelligence", keeping "Copilot".
**Rationale:** Avoids brand confusion in a privacy-positioned product; "Live Assistant" is descriptive and pairs with the existing "Meeting Intelligence" naming.

## [2026-07-06] Local-first model policy for the Live Assistant
**Context:** Privacy is important but not a hard product line; user runs LM Studio with hardware for up to 14B models. Research (2026-07-06) tiered local models and ranked API privacy postures.
**Decision:** Live Assistant defaults to the user-configured local provider (LM Studio, Qwen3-14B class recommended) via the existing `resolveTaskModel` per-task routing — no new provider infra in Phase A. Cloud fallback stays user-chosen per task; future work may add Mistral/Groq and provider privacy badges.
**Alternatives:** Dual-model local tiering now (4B triage + 14B chat — deferred to Phase B where the triage loop exists); adding privacy-ranked cloud providers now (deferred, tracked in STATE.md pending items).
**Rationale:** Phase A is on-demand only, so one 14B model suffices; the per-task routing already expresses "local by default, escalate by choice". Research: complex KG relationship extraction (Phase C) exceeds 14B-class local models — revisit there.

## [2026-07-06] Live transcript state moves to recordingStore; drawer is a global portal
**Context:** The only `onTranscriptSegment` subscriber is MeetingsModern, so live segments are invisible on other views; child overlays hit a known stacking-context trap (standup picker precedent).
**Decision:** Single store-level transcript subscription (recordingStore) consumed by both MeetingsModern and the new LiveMeetingDrawer; drawer renders via `createPortal(document.body)` from AppLayout, only while recording.
**Alternatives:** Dedicated live-meeting route (rejected by user — leaves the Kanban); per-component IPC subscriptions (rejected — duplicate listeners, lost segments off-view).
**Rationale:** The user's core ask is working in Kanban while the meeting runs; one subscription avoids drift between surfaces.

## [2026-07-06] Live Mode pivot — recording becomes a full-screen takeover (supersedes the drawer decision)
**Context:** After LIVE.1 implementation, the user rejected the drawer positioning: "while the recording is happening nothing else should count at that moment... the live assistant should perform all tasks for me." The drawer optimized for multitasking the target user doesn't do mid-meeting.
**Decision:** LIVE.2 replaces the drawer with `LiveModeOverlay` — full-screen (FocusOverlay pattern), auto-enters on recording start, transcript spine + proposals feed + chat. Minimize collapses to the recording pill (deliberate escape hatch, not a hard lock; Esc minimizes, never stops). Drawer shell retired; its content components (transcript feed, chat) are recomposed inside the overlay. recordingStore transcript state carries over unchanged.
**Alternatives:** Hard navigation lock during recording (rejected — when the assistant can't do something, a locked UI turns a small gap into rage); keeping drawer as a middle state (rejected — three UI states is too many).
**Rationale:** For a meeting-intelligence-first product, the meeting is the main event; "LifeDash becomes the meeting" is the stronger product statement, and the expanded toolset makes leaving unnecessary rather than forbidden.

## [2026-07-06] Proactive proposals: propose → one-tap accept (not auto-execute)
**Context:** LIVE.2 pulls the ambient triage loop (old Phase B) forward. MEET-INTEL.1 precedent favored auto-execute+undo, but that was post-meeting batch output of a proven extraction; live triage runs a local 14B on noisy in-flight transcript.
**Decision:** Detected items (action_item/decision/question) land in `live_suggestions` as chips with one-tap Accept/Dismiss. Accepting an action_item creates the card through the live-assistant rails; decisions/questions become brief context. Auto-execute may graduate later as a settings flip. Accepted live items are suppressed from the post-meeting extractor to prevent duplicate cards.
**Alternatives:** Auto-execute + undo (rejected for v1 — noisy transcript + unproven local-model precision could spray the Inbox); hybrid by confidence (rejected — local-model confidence is poorly calibrated; two code paths for v1).
**Rationale:** Trust graduates with evidence; the propose model costs one tap and protects the Inbox while 14B precision is unproven.

## [2026-07-06] One local model serves chat + triage, chat priority (task type `live_triage`)
**Context:** User hardware runs one ~14B model in LM Studio alongside whisper.cpp; LM Studio serializes concurrent requests to a loaded model.
**Decision:** `live_triage` is a distinct task type whose model resolution falls back to `live_assistant`'s — same model by default, splittable in Settings later. Triage is cadence-gated (≥4 new segments ≈ 40s), skips (never queues) when a chat stream is in flight, caps 3 proposals per run, and degrades to "fewer proposals" on JSON failure (retry once, then skip — never throw into the recording pipeline).
**Alternatives:** Loading a second small model (Qwen3-4B) for triage now (rejected — VRAM contention with 14B + whisper unverified; the task-type split keeps this a settings change later); fixed timer instead of segment-count cadence (rejected — ties triage to wall-clock instead of actual speech volume).
**Rationale:** Chat is the user-facing interaction and must stay responsive; triage is background work that tolerates postponement.

## [2026-07-06] LIVE.3 scope + method — field-test fixes and project-proposal triage; Fable plans, Opus ultracode executes
**Context:** First real-meeting test of Live Mode (LIVE.2, uncommitted) surfaced three things: (A) the overlay chat is not clickable (static root-cause hunt across handlers, z-stack, scanlines, ConfirmDialog, the z-[100] pointer-events-none HUD beam, and the 36px TitleBar drag strip found no covering element); (B) high GPU during recording; (C) the user does not want to chat mid-meeting, so the assistant should be able to create a project itself from the conversation.
**Decision:** Bundle all three into a LIVE.3 plan (6 tasks). (1) Clickability fix is EMPIRICAL-FIRST — reproduce via `document.elementsFromPoint` in the running app before touching code; when the fix must be produced headless, apply the justified defensive union (raise the overlay above app chrome z-index + `isolation: isolate` + `WebkitAppRegion:'no-drag'` on the overlay root) and confirm via the user's manual smoke test. (2) GPU is split into inherent-vs-fixable: the dominant cost (whisper.cpp GPU + the 14B LM Studio model + triage cadence) is INHERENT to the 100%-local design and explicitly out of scope; the fixable renderer waste is AudioLevelMeter's never-settling 60fps canvas rAF (Math.random per frame) plus a second meter instance running behind the overlay — park the loop at silence, cap ~30fps, render only one meter; plus a main-process lever: triage yields to in-flight transcription via a one-way injected busy-probe (no new import cycle). (3) Project creation uses the propose→one-tap-accept model, NEVER auto-create: a new `live_suggestions` type `'project'` (+ `acceptedProjectId`, migration 0038); triage proposes at most one "Create project: {name}" chip when the meeting is unlinked; accept creates the project and links the meeting (reusing meetingService.updateMeeting, which already auto-pushes pending action items), rerouting future accepted items into it; a secondary `createProject` agent tool covers the typed-chat path.
**Method:** Planned with Fable (claude-fable-5); executed with Opus 4.8 + ultracode (multi-agent Workflow), per explicit user direction. Tasks run in dependency order (clickability + the two perf tasks are independent; schema 0038 → project rails → triage/UI feature slice must be sequential). Runtime acceptance rests on the user's manual smoke test per project convention.
**Alternatives:** Auto-create + link projects silently (rejected — a 14B on noisy transcript could spawn spurious projects, harder to undo than a card); shotgun z-index changes for the click bug (rejected — guess-fix without evidence); claiming the GPU is "fixable" wholesale (rejected — dishonest; the dominant cost is inherent local inference); post-meeting-only project prompt (rejected — doesn't solve the mid-meeting ask).
**Rationale:** The click bug blocks all other Live Mode verification, so it goes first and must be evidence-driven. Honesty about inherent GPU protects trust while still removing the real renderer waste. The propose→accept project chip extends the already-proven LIVE.2 model with the least new risk and directly answers "I can't chat while people are talking to me."

## [2026-07-07] V3.1 execution — two coexisting boards resolved with an `active` guard; ultracode adversarial review folded into the plan
**Context:** /nexus-execute V3.1 (6 tasks) shipped the session-centric rebuild: 3-entry nav, /session/:id pages (MeetingDetailModal deleted), a `data:changed` broadcast + debounced refetch, EmbeddedBoard extracted from BoardPageModern, the Transcript|Board|Brain live canvas, an activity feed, and query-time FTS. A 22-agent adversarial review (6 dimensions × 2-lens verify) over the full diff found 7 real, V3.1-introduced bugs that 766 green tests + tsc + lint all missed.
**Decision:** (1) **Board coexistence model = an `active` guard on `useBoardController`.** Because `LiveModeOverlay` is a `document.body` portal that layers over the still-mounted route, an overlay-owned EmbeddedBoard and a route-owned EmbeddedBoard can be mounted at once, both driving the single global `boardStore` and both registering document-scoped drag monitors → wrong-board display + duplicate drag side-effects. Resolved by making a covered route board inert: `active={!(isRecording && !liveModeMinimized)}` on route boards (BoardPageModern + SessionWorkspace Board tab); the overlay board stays `active`. An inert controller registers no monitors and does not `loadBoard` (no stomp), and self-heals (reloads its projectId) when `active` flips back true. The overlay-visible predicate `isRecording && !liveModeMinimized` is the canonical "is the live takeover covering the route" signal — reuse it for any future surface that coexists with the overlay. (2) **`useBoardLiveSync` refetch re-reads the current board at timer-fire** (not the id captured at event time) so navigation within the 300ms debounce never resurrects the old board. (3) **The agent `moveCard` tool now broadcasts** — every meeting-agent tool write path hits `notifyDataChanged`. (4) **Per-card Reject/Keep for auto-pushed cards lives on the session Board tab** (real KanbanCardModern cards), NOT reimplemented on the page — there is no "list a meeting's cards" IPC and building one to fake it was rejected as fabrication.
**Alternatives:** Per-project keying of `boardStore` (rejected — a larger, riskier re-architecture than the contained `active` guard; the guard fully addresses both the stomp and the duplicate-monitor findings); scoping drag monitors by `canMonitor` board-id (rejected — the `active` guard is simpler and also fixes the load-stomp, which a canMonitor scope would not); deferring the board-concurrency findings to ISSUES.md (rejected under ultracode — they degrade the headline "watch the assistant work" flow, and the fix proved low-risk with a regression suite).
**Rationale:** The overlay-over-route topology is intrinsic to the full-screen live takeover, so the coexistence had to be modelled explicitly rather than assumed away; the `active` guard is invisible to users (an inert board is always fully covered) yet deterministically prevents the store stomp and duplicate side-effects. Running the adversarial review before the user's smoke test caught defects (infinite-spinner dead-end, wrong-board race, onboarding tour walking new users through deleted sections) that no automated gate surfaced — validating the ultracode "verify before you trust green" posture on a large, MEDIUM-confidence diff.

## [2026-07-07] Brain live growth: refetch-and-diff, not event→insert (V3.2 promotion revalidation)
**Context:** Promoting PLAN-V3.2 into PLAN.md. The parked Task 4 mapped renderer events to targeted mind-map inserts ("new card → insert under its column node"), with dedupe + a >25-delta refetch fallback. Revalidation against the shipped V3.1 code showed the premise is false: the `data:changed` payload (dataChangeNotifier.ts) is `{ scope: 'cards'|'columns'|'projects', projectId? }` — scope-level by design, no entity identity — so the renderer cannot know WHAT changed, only that something did.
**Decision:** brainStore refetches the active scope's full tree (300ms trailing debounce on `data:changed`, mirroring useBoardLiveSync) and DIFFS node-id sets (ids stable, derived from entity type + id) to find entering nodes — those drive entrance animations, collapsed-ancestor chevron badges, and the Brain canvas badge. Decision/question chip accepts, which never emit `data:changed` (no card mutation), trigger the same refresh through the liveSuggestionsStore accept() success seam — the exact hook that already feeds activityFeedStore. The parked dedupe/threshold machinery is deleted.
**Alternatives:** Enriching `data:changed` payloads with entity ids + parent paths so targeted inserts work (rejected — adds main-process surface and a second consumer contract to keep honest at every mutation site, for marginal gain when the full tree is hundreds-to-low-thousands of nodes over one IPC); keeping event→insert only for the suggestion-accept path where rich data exists (rejected — two growth code paths that must agree beats nothing; one path that cannot drift wins).
**Rationale:** Refetch-and-diff is one code path for every trigger, idempotent under the broadcaster's known double-fires (they collapse into a single debounced refetch; an unchanged id-set animates nothing), and strictly simpler than what it replaces — the scope-level broadcast was V3.1's deliberate "impossible to forget at new mutation sites" design, and this keeps V3.2 a pure consumer of it.

## [2026-07-08] V3.3.5 execution — parallel-batch contract-freeze; merge-not-replace refine seed (data-loss fix); provider-native web search (supported); Gemini as frontier provider
**Context:** /nexus-execute V3.3.5 (6 tasks, 989→1126 tests). Executed as a parallel batch and hardened by a 23-agent adversarial review. Several forks and one serious defect were resolved during execution.
**Decision:**
(1) **Deep-creation drafts MERGE over the existing profile on refine; they never replace it.** `TwinWizard.seedFromPanelDraft` seeds the shared editable review from a deep/history/web `Partial<TwinProfileSections>` draft via `seedSuppliedSections(prev, draft)`, which overrides ONLY the sections the draft actually supplies (`'key' in draft`) and preserves every section it omits. This closes a HIGH-severity data-loss bug the adversarial review found (3 dimensions, both verify lenses): because `twinProfileService.updateProfileSection` is a per-section jsonb REPLACE and the wizard's `writeAllSections` writes all 8 sections on save, the previous "reseed drafts from the panel draft alone" wiped stored `identity`/`preferences` (which history mining structurally excludes) with `{}`. Guarded by `TwinWizard.refineMerge.test.tsx` (failing-before/passing-after). `brief` remains special-cased.
(2) **The deep interview lives in a NEW `twinDeepInterviewService` (interviewNext/interviewSynthesize), not `twinInterviewService`** — keeps the V3.3 `draftSection` seam untouched and gives the frozen IPC a stable delegate. Transport is invoke-per-turn (extends the V3.3 invoke-not-streaming decision): each turn is a small request→JSON via `taskType:'twin_interview'`, validate-retry-skip, hard 8-question cap.
(3) **Web research uses provider-native server-side search ONLY, and the installed stack SUPPORTS it** — the Task-4 preflight (reading node_modules types, not memory) confirmed `ai@6` + `@ai-sdk/{anthropic,openai,google}` expose `webSearch_20250305`/`webSearch`/`googleSearch` with citations on `steps[].sources`. Capability is gated on `twin:get-creation-model.isFrontier`; local/no-model providers render an honest absence. No scraping, no new search-API key surface (the STOP-condition was not triggered).
(4) **Google Gemini is a first-class frontier provider**, wired exactly like OpenAI/Anthropic. `@ai-sdk/google` is pinned to `^3.0.90` (provider-spec-v3, compatible with `ai@6`) — NOT 4.x, which pulls provider-v4 and mismatches the rest of the SDK. `'google'` was added to `AIProviderName` and every exhaustive `Record<AIProviderName,…>` map + the IPC validation schema; `FRONTIER_PROVIDERS = openai|anthropic|google` is the single definition of "frontier".
(5) **Cloud-routed history mining requires a per-run consent dialog (no remember-me); the consent descriptor is computed with ZERO model calls; and only what the descriptor discloses is sent** — the "People who spoke" speaker block was removed from the miner context (it carried only generic "Speaker N" diarizer labels, never real names) so disclosed==sent, closing an adversarial-review consent-copy gap.
(6) **Execution methodology: contract-freeze parallel batch.** Task 1 created/froze every shared file (schema, `src/shared/types/twin.ts`, 6 IPC channels, `ai.ts` frontier types, and honest stub files/components with their mount points), after which Tasks 2/3/4/5 ran as a file-disjoint concurrent batch editing only files they own, with a STOP-and-report seam (no sibling edits to shared files) and tree-wide gates run only post-batch. A follow-up round solidified the draft→review seam BEFORE the batch spawned, preventing a mid-batch double-STOP.
**Alternatives:** Append/dedupe list sections on refine instead of override-supplied (rejected for now — override-supplied is the minimal correct data-loss fix; the user reviews before save, and list-merge semantics can be a later refinement); polyfilling web search via scraping/a new search key if unsupported (pre-authorized STOP path — not needed since native search is present); adding `'google'` to `AIProviderName` in Task 1 (rejected — it would have broken Task 1's own exhaustive-map tsc gate; deferred to Task 5 which owns those maps); one big sequential execution (rejected — the batch is file-disjoint by construction, so concurrency is safe and much faster); counting speakers in the consent dialog instead of removing the block (rejected — the block is useless generic labels, so removal both closes the disclosure gap and drops dead prompt tokens).
**Rationale:** The refine merge preserves the project's non-negotiable "never delete local data" sync-safety principle at the one place V3.3.5 could have violated it; the batch methodology got four features landed collision-free by making Task 1's frozen contracts the only cross-agent coupling; and the adversarial review (which found the data-loss bug that 1118 green tests, tsc, and eslint all missed) is the same "verify before you trust green" discipline that paid off on V3.1 — here it also pruned an overstated privacy finding to its true minor form.

## [2026-07-10] V3.4 execution — LM Studio echoes the real embedding model (provenance from response.model, not the requested id); forgotten facts gate re-learning; card-dominant Ask answers suppress session citations
**Context:** /nexus-execute V3.4 (the FINAL v3 phase — twin learning loop + local semantic layer; 6 tasks, 1154→1279 tests). Executed as preflight+contract-freeze (Task 1) + a `search:ask` seam follow-up → two file-disjoint parallel batches (2+4, then 3+5) → a 28-agent adversarial review (8 dims × 2-lens, 10 raw findings → 6 survived) → a dedicated fix pass → Task 6 integration. Several execution-time discoveries and forks were resolved.
**Decision:**
(1) **Embedding-model provenance is the provider-ECHOED `response.model`, never the requested id.** The user-assisted preflight discovered LM Studio's `/v1/embeddings` silently routes to whatever embedding model is loaded and ignores a requested id that isn't a loaded embedding model (proven: requesting the chat model `gemma-4-12b-qat` returned nomic-v1.5's vectors with `response.model = nomic`). So `embed()` surfaces the echoed model, `embedding_index_meta` records it, and BOTH the write-side and read-side mismatch guards compare the echoed model — otherwise the "index built with X, now Y → rebuild" affordance is blind. `EMBEDDING_DIM=768` was locked by live MEASUREMENT of EmbeddingGemma-300M (multilingual), not from research notes.
(2) **`resolveTaskModel('embedding')` returns null when unconfigured (no silent cloud fallback for BULK content), and the privacy hint is provider-aware at the point of choice.** An adversarial finding caught the embedding Settings hint asserting "stays on your device" even when a cloud provider was selected — fixed to mirror the `live_assistant` provider-gated hint (amber warning for cloud). No-silent-cloud made honest at the decision surface.
(3) **Forget is DURABLE: fact-extraction dedupe filters candidates against ACTIVE *and* FORGOTTEN facts**, but forgotten content is never re-disclosed to the model (post-generation filter, not a prompt exclusion). An adversarial finding showed dedupe-vs-active-only silently resurrected a forgotten fact when a later session re-mentioned it — defeating the safety-triad Forget control.
(4) **Read-side embedding-model mismatch records the rebuild flag.** `retrieveVectorHits` returning `[]` on mismatch (correct — never mix vector spaces) previously surfaced no affordance on a fully-indexed static corpus; it now persists the same `embedding.indexMismatch` setting the write path uses, via a helper exported from embeddingService.
(5) **Ask citations = cite-what-was-retrieved (a deliberate RAG convention), with one guard:** when the single closest grounding chunk is a card, session citations are suppressed (a card-dominant answer must not attribute to co-retrieved sessions). The fragile alternative (LLM emitting `[n]` usage indices) was rejected. The no-answer sentinel match was hardened to fold typographic apostrophes.
(6) **Card embeddings store `meetingId=null`** (not `sourceMeetingId`) so the `embeddings.meetingId` ON DELETE cascade cannot drop a surviving card's embedding when its origin meeting is deleted.
(7) **Post-session ordering (facts → entities) is guaranteed by ES-module import evaluation**, not boot-file ordering: `entityService` imports `isLearningPaused` from `twinMemoryService`, so twinMemoryService's top-level hook self-registration runs first. Entity extraction rides `twin_learning` routing + the pause gate and is capped flat (≤8 person/topic entities/session, dedupe by normalizedName); NO entity↔entity relationships in v3 (the 14B relationship-extraction cliff — a hard ceiling).
**Alternatives:** Trusting the requested embedding-model id for provenance (rejected — LM Studio proves it lies); a hard schema dimension from research (rejected — measured 768 tolerates model swaps); dedupe forgotten facts by re-adding them to the prompt exclusion list (rejected — would re-disclose forgotten content to the model; a post-generation filter keeps them out of sight); LLM `[n]`-index citations (rejected — model-dependent/fabrication-prone, per the refuting verifier); folding the review fixes into Task 6 (rejected — a dedicated fix pass kept each finding focused with a fail-before/pass-after test before integration).
**Rationale:** The phase's two hard assumptions (pgvector-in-PGlite, live local embeddings) were converted to evidence before any schema commit, and the preflight itself surfaced the LM Studio routing behavior that reshaped provenance. Contract-freeze made two 2-agent concurrent batches collision-free; the one missing seam (`search:ask`) was patched via a Task-1 follow-up, never a sibling edit. The 28-agent adversarial review caught 6 real defects (3 MED privacy/consent/correctness) that 1252 green tests missed while its 2-lens verification refuted 4 non-issues (incl. a plausible queue lost-wakeup proven unreachable in the single-threaded event loop) — the same "verify before you trust green" discipline that paid off on V3.1 and V3.3.5, now closing out the v3 program.
