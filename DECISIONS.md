# Decisions Register (append-only)

## [2026-07-06] Live-agent initiative ‚Äî three-phase direction (LIVE.1 = Phase A)
**Context:** User wants LifeDash to shift from post-hoc pipeline (record ‚Üí review later) to a live experience: an AI partner during recording, evolving toward Graphify-style structured knowledge (entities + typed edges + confidence tags, queryable by the agent).
**Decision:** Three sequential phases: A) Live Assistant (drawer with live transcript + on-demand agent chat with tools), B) ambient auto-triage feed (proactive suggestions, one-tap accept), C) knowledge-graph substrate (entity/edge tables in PGlite, extraction post-meeting first). LIVE.1 covers A only.
**Alternatives:** Building the knowledge graph first (rejected ‚Äî most experimental, needs the live surface to be useful); one mega-plan for all three (rejected ‚Äî exceeds 6-task limit, bundles unrelated smoke tests).
**Rationale:** Each phase validates the next; A reuses proven infra (streaming transcription, card-agent tool loop) so it's the cheapest slice that changes the product's feel.

## [2026-07-06] "Live Assistant" naming (not "Copilot")
**Context:** User asked whether "copilot" meant Microsoft Copilot ‚Äî the term is now trademark-loaded and could read as a Microsoft integration.
**Decision:** User-facing name "Live Assistant"; internal naming `meeting_agent` (schema, service, IPC), task type `live_assistant`.
**Alternatives:** "Meeting Partner", "Live Intelligence", keeping "Copilot".
**Rationale:** Avoids brand confusion in a privacy-positioned product; "Live Assistant" is descriptive and pairs with the existing "Meeting Intelligence" naming.

## [2026-07-06] Local-first model policy for the Live Assistant
**Context:** Privacy is important but not a hard product line; user runs LM Studio with hardware for up to 14B models. Research (2026-07-06) tiered local models and ranked API privacy postures.
**Decision:** Live Assistant defaults to the user-configured local provider (LM Studio, Qwen3-14B class recommended) via the existing `resolveTaskModel` per-task routing ‚Äî no new provider infra in Phase A. Cloud fallback stays user-chosen per task; future work may add Mistral/Groq and provider privacy badges.
**Alternatives:** Dual-model local tiering now (4B triage + 14B chat ‚Äî deferred to Phase B where the triage loop exists); adding privacy-ranked cloud providers now (deferred, tracked in STATE.md pending items).
**Rationale:** Phase A is on-demand only, so one 14B model suffices; the per-task routing already expresses "local by default, escalate by choice". Research: complex KG relationship extraction (Phase C) exceeds 14B-class local models ‚Äî revisit there.

## [2026-07-06] Live transcript state moves to recordingStore; drawer is a global portal
**Context:** The only `onTranscriptSegment` subscriber is MeetingsModern, so live segments are invisible on other views; child overlays hit a known stacking-context trap (standup picker precedent).
**Decision:** Single store-level transcript subscription (recordingStore) consumed by both MeetingsModern and the new LiveMeetingDrawer; drawer renders via `createPortal(document.body)` from AppLayout, only while recording.
**Alternatives:** Dedicated live-meeting route (rejected by user ‚Äî leaves the Kanban); per-component IPC subscriptions (rejected ‚Äî duplicate listeners, lost segments off-view).
**Rationale:** The user's core ask is working in Kanban while the meeting runs; one subscription avoids drift between surfaces.

## [2026-07-06] Live Mode pivot ‚Äî recording becomes a full-screen takeover (supersedes the drawer decision)
**Context:** After LIVE.1 implementation, the user rejected the drawer positioning: "while the recording is happening nothing else should count at that moment... the live assistant should perform all tasks for me." The drawer optimized for multitasking the target user doesn't do mid-meeting.
**Decision:** LIVE.2 replaces the drawer with `LiveModeOverlay` ‚Äî full-screen (FocusOverlay pattern), auto-enters on recording start, transcript spine + proposals feed + chat. Minimize collapses to the recording pill (deliberate escape hatch, not a hard lock; Esc minimizes, never stops). Drawer shell retired; its content components (transcript feed, chat) are recomposed inside the overlay. recordingStore transcript state carries over unchanged.
**Alternatives:** Hard navigation lock during recording (rejected ‚Äî when the assistant can't do something, a locked UI turns a small gap into rage); keeping drawer as a middle state (rejected ‚Äî three UI states is too many).
**Rationale:** For a meeting-intelligence-first product, the meeting is the main event; "LifeDash becomes the meeting" is the stronger product statement, and the expanded toolset makes leaving unnecessary rather than forbidden.

## [2026-07-06] Proactive proposals: propose ‚Üí one-tap accept (not auto-execute)
**Context:** LIVE.2 pulls the ambient triage loop (old Phase B) forward. MEET-INTEL.1 precedent favored auto-execute+undo, but that was post-meeting batch output of a proven extraction; live triage runs a local 14B on noisy in-flight transcript.
**Decision:** Detected items (action_item/decision/question) land in `live_suggestions` as chips with one-tap Accept/Dismiss. Accepting an action_item creates the card through the live-assistant rails; decisions/questions become brief context. Auto-execute may graduate later as a settings flip. Accepted live items are suppressed from the post-meeting extractor to prevent duplicate cards.
**Alternatives:** Auto-execute + undo (rejected for v1 ‚Äî noisy transcript + unproven local-model precision could spray the Inbox); hybrid by confidence (rejected ‚Äî local-model confidence is poorly calibrated; two code paths for v1).
**Rationale:** Trust graduates with evidence; the propose model costs one tap and protects the Inbox while 14B precision is unproven.

## [2026-07-06] One local model serves chat + triage, chat priority (task type `live_triage`)
**Context:** User hardware runs one ~14B model in LM Studio alongside whisper.cpp; LM Studio serializes concurrent requests to a loaded model.
**Decision:** `live_triage` is a distinct task type whose model resolution falls back to `live_assistant`'s ‚Äî same model by default, splittable in Settings later. Triage is cadence-gated (‚â•4 new segments ‚âà 40s), skips (never queues) when a chat stream is in flight, caps 3 proposals per run, and degrades to "fewer proposals" on JSON failure (retry once, then skip ‚Äî never throw into the recording pipeline).
**Alternatives:** Loading a second small model (Qwen3-4B) for triage now (rejected ‚Äî VRAM contention with 14B + whisper unverified; the task-type split keeps this a settings change later); fixed timer instead of segment-count cadence (rejected ‚Äî ties triage to wall-clock instead of actual speech volume).
**Rationale:** Chat is the user-facing interaction and must stay responsive; triage is background work that tolerates postponement.

## [2026-07-06] LIVE.2 close-out (Task 6): shared chip rendering, prop-driven post-meeting load, real-pipeline anti-dup test
**Context:** Final LIVE.2 task needed a post-meeting surface for un-actioned proposals, the `'live-assistant'` provenance badge, release notes, and a test proving the accepted-live-item anti-dup contract (Task 2) actually prevents a duplicate card, not just a prompt string.
**Decision:** (1) Extracted the chip/confirmation markup out of `LiveProposalsFeed` into a new presentational `LiveProposalChips` component; `LiveProposalsFeed` (store-driven, live) and the new `meeting-detail/LiveProposalsSection` (prop-driven, loads via `listLiveSuggestions` independently of `liveSuggestionsStore` since that store clears on stop) both render through it ‚Äî one markup, two data sources. (2) `'live-assistant'` added to the `CardSource` union; its `KanbanCardModern` badge mirrors `'auto-from-meeting'`'s structure exactly (badge + tooltip + `handleViewSource` click) but is a separate conditional block ‚Äî it intentionally does NOT gate the Reject/Keep-in-Inbox overflow menu, since that menu is specific to the auto-push review flow and doesn't apply to live-assistant cards. (3) The anti-dup integration test does NOT mock `autoPushService` ‚Äî it runs the real `generateActionItems` ‚Üí `autoPushActionItems` round trip against a table-identity-routed DB double, asserting on the actual `cards` insert call (exactly one card, not matching the suppressed title) rather than only on the LLM prompt string. This is a stronger proof than Task 2's own tests, but still assumes LLM compliance with the suppression instruction ‚Äî there is no code-level title de-dup filter, by original Task 2 design. (4) Corrected the stale "Live Assistant drawer" release-note bullet (pre-dated the Task 4 full-screen pivot) to describe Live Mode instead of leaving factually wrong un-released copy.
**Alternatives:** A wholly separate hand-rolled list component for the meeting-detail section (rejected ‚Äî direct markup duplication, against the story's "reuse, don't rebuild" instruction); keeping the anti-dup test mocked at the `autoPushActionItems` call-args level only (rejected ‚Äî proves plumbing, not that a duplicate card is actually withheld); adding a code-level title-matching filter in `generateActionItems` to make suppression a hard guarantee (rejected ‚Äî out of scope for Task 6, which only adds tests/UI per the story; would be a Task 2 design change requiring its own sign-off).
**Rationale:** Reuse over rebuild kept the two proposal surfaces from drifting apart; asserting on real DB-insert calls makes the anti-dup test meaningful rather than tautological; leaving the prompt-only suppression mechanism alone respects the already-locked Task 2 contract instead of silently expanding this task's scope.

## [2026-07-06] LIVE.3 scope + method ‚Äî field-test fixes and project-proposal triage; Fable plans, Opus ultracode executes
**Context:** First real-meeting test of Live Mode (LIVE.2, uncommitted) surfaced three things: (A) the overlay chat is not clickable (static root-cause hunt across handlers, z-stack, scanlines, ConfirmDialog, the z-[100] pointer-events-none HUD beam, and the 36px TitleBar drag strip found no covering element); (B) high GPU during recording; (C) the user does not want to chat mid-meeting, so the assistant should be able to create a project itself from the conversation.
**Decision:** Bundle all three into a LIVE.3 plan (6 tasks). (1) Clickability fix is EMPIRICAL-FIRST ‚Äî reproduce via `document.elementsFromPoint` in the running app before touching code; when the fix must be produced headless, apply the justified defensive union (raise the overlay above app chrome z-index + `isolation: isolate` + `WebkitAppRegion:'no-drag'` on the overlay root) and confirm via the user's manual smoke test. (2) GPU is split into inherent-vs-fixable: the dominant cost (whisper.cpp GPU + the 14B LM Studio model + triage cadence) is INHERENT to the 100%-local design and explicitly out of scope; the fixable renderer waste is AudioLevelMeter's never-settling 60fps canvas rAF (Math.random per frame) plus a second meter instance running behind the overlay ‚Äî park the loop at silence, cap ~30fps, render only one meter; plus a main-process lever: triage yields to in-flight transcription via a one-way injected busy-probe (no new import cycle). (3) Project creation uses the propose‚Üíone-tap-accept model, NEVER auto-create: a new `live_suggestions` type `'project'` (+ `acceptedProjectId`, migration 0038); triage proposes at most one "Create project: {name}" chip when the meeting is unlinked; accept creates the project and links the meeting (reusing meetingService.updateMeeting, which already auto-pushes pending action items), rerouting future accepted items into it; a secondary `createProject` agent tool covers the typed-chat path.
**Method:** Planned with Fable (claude-fable-5); executed with Opus 4.8 + ultracode (multi-agent Workflow), per explicit user direction. Tasks run in dependency order (clickability + the two perf tasks are independent; schema 0038 ‚Üí project rails ‚Üí triage/UI feature slice must be sequential). Runtime acceptance rests on the user's manual smoke test per project convention.
**Alternatives:** Auto-create + link projects silently (rejected ‚Äî a 14B on noisy transcript could spawn spurious projects, harder to undo than a card); shotgun z-index changes for the click bug (rejected ‚Äî guess-fix without evidence); claiming the GPU is "fixable" wholesale (rejected ‚Äî dishonest; the dominant cost is inherent local inference); post-meeting-only project prompt (rejected ‚Äî doesn't solve the mid-meeting ask).
**Rationale:** The click bug blocks all other Live Mode verification, so it goes first and must be evidence-driven. Honesty about inherent GPU protects trust while still removing the real renderer waste. The propose‚Üíaccept project chip extends the already-proven LIVE.2 model with the least new risk and directly answers "I can't chat while people are talking to me."
