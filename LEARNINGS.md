## 2026-07-21 | process-incident | node_modules, git worktree
**Issue:** A `git worktree` was created at an old commit for an exact baseline eslint comparison, with an NTFS *junction* linking its `node_modules` to the main project's real `node_modules` (to avoid a second `npm install`). Running `git worktree remove --force` on that worktree caused Windows' recursive delete to follow the junction transparently and wipe the main project's actual `node_modules` contents.
**Context:** GUARD.1 Task 6 (integration/gates), diagnosing a 3-warning eslint drift by diffing against the pre-phase commit `b511eda`.
**Fix:** Recovered via `npm ci` (package-lock.json was intact, no tracked source files were touched since node_modules is gitignored). Going forward: never link or junction `node_modules` (or any real, non-disposable directory) into a temporary `git worktree` on Windows — `git worktree remove` does not treat a junction as a boundary and will recursively delete through it. For a clean baseline comparison against an old commit, either run a full separate `npm install` inside the worktree, or avoid the worktree entirely and diff via `git show <rev>:<path>` / `git diff <rev> -- <path>`.
**Tags:** windows, git-worktree, node_modules, destructive-cleanup, cross-platform
